## Evidence

### Classic "wait for any of N events" pattern (rank=high)

The function busy-polls 4 BIOS event handles in a fixed order and
returns the index of the first one that fires (1..4). The
`bios_TestEvent` PsyQ helper returns 1 when an event is in the
"signaled" state. The function structure -- spin until any of the
tested handles signals, return its 1-based index -- is the canonical
"wait for the first of N events" pattern (the K&R-era equivalent of
`select()` over a 4-handle event set).

This is unambiguous: there is no other meaningful behaviour for
`while (true) { for h in handles: if TestEvent(h) == 1: return idx }`.

### The 4 event handles match the partner shutdown's set (rank=medium)

The 4 handles `D_800A3838` / `D_800A383C` / `D_800A3848` / `D_800A3850`
are tested here and also closed by
[`func_80037774`](func_80037774.md) (proposed
`game_event_shutdown_*`). So this function consumes the same event
pool that shutdown later releases.

### `s32 one = 1;` is a maspsx codegen artifact (rank=info)

The `s32 one = 1;` local with subsequent `== one` compares (rather
than the obvious `== 1`) is a GCC 2.7.2 / maspsx code-shape that
forces `$one` into a register so the per-iteration comparison
doesn't need to re-li the immediate. Idiomatic for tight polling
loops at PSX SDK era. The C-side oddity is *not* functional; it
just biases the register allocator.
