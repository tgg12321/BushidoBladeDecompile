## Evidence

### Calls `disp_SetFramebufferMode` (rank=high)

The second statement is a direct call to an already-named function in
the display subsystem -- `disp_SetFramebufferMode(mode, r, g, b)`. The
specific argument tuple `(1, 0, 0, 0)` means "switch to mode 1 with
the clear colour set to (0,0,0)" -- a standard
"reset-display-to-default" call.

This anchors the function firmly in the display-init subsystem.

### `D_800A35E4 = 0` is a per-frame-state reset (rank=medium)

Setting a global byte/word to 0 right after switching display modes
is the standard "reset transition-progress / fade-state / dirty-flag"
pattern. This is consistent with the function being an "I'm about to
draw with mode N -- initialise everything for that mode" helper.

### `func_80068F70(a0, &D_8009BD24)` (rank=medium)

The first call passes `a0` and a fixed config block (`D_8009BD24`) to
`func_80068F70` -- a 2-arg "load setup data" pattern. The fact that
`a0` is the *same* mode argument that gets used downstream by
`disp_SetFramebufferMode` (indirectly -- the latter receives a fixed
1, not `a0`) suggests `a0` is selecting which sub-mode's config to
load from `D_8009BD24`.

So the overall shape is: "for sub-mode `a0`, load its config,
reinitialise the framebuffer, clear the per-mode flag, signal
success".

## Why this name

* `disp_*` follows the existing display-subsystem naming convention
  (e.g., `disp_SetFramebufferMode`, `disp_*` helpers in display.c).
* `init_for_mode` reflects the call sequence: load config, switch
  framebuffer mode, reset per-mode flag.
* Address suffix retained because (a) "init" vs "switch" vs
  "transition" is not fully pinned down without the call-graph trace
  of `func_80068F70`, (b) we don't know if this is the *only* such
  helper or one of several mode-specific variants.

## Caller-side audit (2026-05-17 reinforcement) -- CORRECTS original analysis

**Both callers pass a memory ADDRESS, not a small integer mode:**

* `func_80077820((s32)0x80118800)` (code6cac_b2.c:144) -- after
  `D_800A31D8 = v0; func_80035438();`
* `func_80077820((s32)0x80118800)` (code6cac_b2.c:292) -- after
  `file_ResetDmaFlag(); obj_InitAll();`

`0x80118800` is a fixed RAM address in the upper region (~1.1 MB into
RAM) that the game uses as a fixed-location buffer for streamed data.
Both call sites pass the *same* address; the function is therefore
not a "mode selector" but a **config loader that reads from a fixed
upper-RAM buffer**.

Re-reading the body with this in mind:

```c
s32 func_80077820(s32 a0) {
    func_80068F70(a0, (s32 *)&D_8009BD24);   // copy from a0 to D_8009BD24
    disp_SetFramebufferMode(1, 0, 0, 0);     // unconditionally to mode 1
    D_800A35E4 = 0;
    return 1;
}
```

The original "init for mode" reading was wrong -- the
`disp_SetFramebufferMode(1, 0, 0, 0)` call is *unconditional* (always
mode 1, not parametrised by a0). a0 is the *source* of a config-copy
operation, not a mode selector.

**Refined name:** `disp_load_config_from_buf_80077820` --
"load the display configuration from buffer at `a0` into `D_8009BD24`,
then reset to display mode 1." The slug is more honest about what
the function does.
