## Evidence

### Two sequential table-scan loops (rank=high)

The function consists of two consecutive `do { ... } while` loops,
each iterating a fixed-stride table:

**Loop 1** -- 32 entries at base `D_80101BF0`, stride 16 bytes:

* Reads byte at slot start (`*p`)
* If it matches `D_800A36F8` (some global key value), overwrites
  with `0xFF` (the "free" marker).
* This is the "**release any slot matching this key**" pattern.

**Loop 2** -- N entries at base `D_800F68E0`, stride 16 bytes:

* Reads halfword at slot start (`*q`)
* If not `-1` (i.e., slot is active), increments it by 1.
* If the new value exceeds `*(u8 *)(q + 2) + 0x101` (i.e., the slot's
  per-slot timeout limit + 0x101), sets it to `-1` (mark expired).

So loop 2 is a **per-frame tick of a timer table**: each active
slot's counter is bumped, and slots whose counters exceed the
slot-specific threshold are released. Loop 1 is a separate
"release-by-key" pass that runs first.

Together: this is a per-frame maintenance hook that

1. Releases any timer slot keyed by `D_800A36F8` (some event
   trigger).
2. Ticks every active timer slot's counter and expires those past
   their deadline.

### `0x10` stride implies a per-slot struct of 16 bytes (rank=medium)

Both loops use a 16-byte stride. Loop 1 reads only byte 0 of the
slot; loop 2 reads halfword 0 (counter) and byte 2 (timeout limit).
A reasonable per-slot layout:

```
+0x00  s16 counter       (or u8 key + u8 padding)
+0x02  u8  timeout_limit
+0x03  u8  ???
+0x04..0x0F  payload    (event data, callback ptr, etc.)
```

### Two separate tables, but same stride and same tick (rank=info)

`D_80101BF0` (loop 1) and `D_800F68E0` (loop 2) are completely
different memory regions, but they share the 16-byte slot stride.
Possibly they're two pools (e.g., short-lived vs persistent timers,
or per-character vs global timers).
