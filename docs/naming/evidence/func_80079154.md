## Body

```c
    D_800F1848 = D_800F1848 * 0x41C64E6D + 0x3039;
    return (D_800F1848 >> 16) & 0x7FFF;

```

## Evidence

### Constants are the canonical BSD libc LCG (rank=high)

The two magic numbers in the body are the exact constants of the linear
congruential generator used by the classic Brian Kernighan BSD `rand()`
implementation, reused by glibc, PSX SDK, and most pre-2000 C runtimes:

* `0x41C64E6D` = **1103515245** -- the multiplier `a`.
* `0x3039`     = **12345**      -- the additive constant `c`.
* The output transform `(state >> 16) & 0x7FFF` discards the low bits
  (known to have poor randomness in any LCG) and masks to 15 bits.
* The range of the return value is `[0, 32767]`, which is the POSIX
  `RAND_MAX` constant `((1<<15) - 1)`.

The recurrence is therefore `state' = state * 1103515245 + 12345; return
(state' >> 16) & 0x7FFF;` -- byte-for-byte the standard `rand()` defined
by `man 3 rand` on the same generation of Unix systems that the PsyQ
SDK (1993--1997) was targeting.

These constants are not load-bearing for any other algorithm we have
seen in the BB2 binary, and the function's single state variable
`D_800F1848` matches the libc convention of a `static unsigned long
next` seed at file scope. The mathematical signature is unique.

### Caller pattern is consistent with a general-purpose RNG (rank=medium)

29 call sites spread across **6 distinct source files** in 5 different
subsystems (general code6cac, character logic, ings menu, text1b
rendering, text1b_b LUT helpers). No two callers share a closely
related stack frame -- they invoke the function in unrelated contexts
ranging from menu animation jitter to per-frame combat dice rolls.
This is consistent with a single shared RNG used throughout the game.

If the function were instead a domain-specific helper (e.g., a
collision-resolution helper) we would expect the calls to cluster
inside one or two subsystems. They do not.

## Why this name

* Matches the project's convention of prefixing PSX-runtime-style
  helpers with `bb2_` (e.g., `bb2_memcpy`).
* `rand` is the standard C name for the function this body
  implements; the prefix disambiguates it from any PsyQ-shipped
  symbol (the PsyQ SDK did include `rand`, but this one is inlined
  into BB2's own binary, not linked from libapi).
* An alternative (`psyq_rand`) would suggest this is a PsyQ stdlib
  copy detected by `known_psyq_stdlib.txt`. It isn't -- the analyzer
  did not flag it. Calling it `bb2_rand` reflects that the body lives
  in BB2's own .text.
