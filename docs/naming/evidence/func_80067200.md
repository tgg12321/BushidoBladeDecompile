## Evidence

**The asm body opens with GTE matrix setup** -- a series of `lw` + `ctc2` instructions reading from `D_800A3474` (the cached matrix data) and writing into GTE cop2 control registers 0..4. These five register slots hold the rotation matrix (ROT11..ROT33 in PSX SDK naming).

**Branches on `$s0 < 4`** at the start, setting `($s6, $s5) = (0x41, 0x3F)` for the smaller branch and `(0x31, 0x1F)` for the larger. The values 0x40 / 0x20 (= 64 / 32) are standard PSX texture-page sub-divisions; the +/-1 versions are inclusive-exclusive boundaries for texture coordinate clamping.

**6 wrappers feed it state** (the `trigger_event_*_8006706x/9x/13x/16x/19x/1CC` family from batch 4): each sets a per-event state slot (`D_800F1128`/`D_800F1130`/`D_800F1134`) to 1 or 2 after calling this with `(event_pair_a, event_pair_b, flag)`. The state slot tracks which variant was last drawn.

## Why this name

GTE-using effect dispatcher that renders a textured-quad (or similar small primitive) under a cached rotation matrix. Confidence **medium** because the GTE setup + the texture-coord constants make the GTE / draw role unambiguous, but the exact primitive being rendered (quad? triangle? tile?) requires reading the full ~458-line asm to confirm.
