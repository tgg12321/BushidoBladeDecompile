## Evidence

### Closes 8 BIOS events inside a critical section (rank=high)

The function takes the canonical "shutdown / unregister" shape:

1. `EnterCriticalSection()` -- prevent interrupts from racing.
2. `bios_CloseEvent(handle)` x 8 -- release every event handle owned
   by this module.
3. `ExitCriticalSection()` -- restore interrupt state.
4. `func_8007A400()` -- a post-shutdown callback (presumably resets
   the per-module state that referred to the now-closed events).

The pattern is unambiguous: this is the **shutdown / cleanup** half of
a per-module BIOS-event lifecycle. There is no other valid
interpretation of "close 8 events under a critical section" -- this
is the inverse of an OpenEvent chain elsewhere.

### The 8 event handles are *the same set* used by the partner waiter (rank=medium)

`func_80037964` (separately documented in
[func_80037964.md](func_80037964.md), proposed
`game_event_wait_any_80037964`) tests 4 of these handles
(`D_800A3838`, `D_800A383C`, `D_800A3848`, `D_800A3850`) in a
`bios_TestEvent` loop. The other 4 are presumably tested elsewhere
in the same module.

So the event handles `D_800A37DC` / `..F0` / `..FC` / `..3800` /
`..3838` / `..383C` / `..3848` / `..3850` are a coherent group --
the per-module event pool. This function unwinds all of them, which
is by definition the module's shutdown hook.

## 2026-05-17 deepen-pass refinement -- OpenEvent owner identified

The 8 events are opened by the already-named function
`pad_press_control` at `src/code6cac_c.c:175` -- but **that name is
itself a misnomer**. The function actually does memory-card init
plus opens 8 memcard events:

```c
void pad_press_control(void) {
    ...
    bios__bu_init_A0();    // !! memcard init -- BU = Backup Unit
    bios_ChangeClearPad(0);
    EnterCriticalSection();
    D_800A37DC = bios_OpenEvent(0xF4000001, 4,      0x2000, 0);
    D_800A37F0 = bios_OpenEvent(0xF4000001, 0x8000, 0x2000, 0);
    D_800A37FC = bios_OpenEvent(0xF4000001, 0x100,  0x2000, 0);
    D_800A3800 = bios_OpenEvent(0xF4000001, 0x2000, 0x2000, 0);
    D_800A3838 = bios_OpenEvent(0xF0000011, 4,      0x2000, 0);
    D_800A383C = bios_OpenEvent(0xF0000011, 0x8000, 0x2000, 0);
    D_800A3848 = bios_OpenEvent(0xF0000011, 0x100,  0x2000, 0);
    D_800A3850 = bios_OpenEvent(0xF0000011, 0x2000, 0x2000, 0);
    ExitCriticalSection();
    // ...8x bios_EnableEvent
}
```

**Findings:**

1. The 8 events are for **memory card I/O**, not generic game
   events. `bios__bu_init_A0()` (BU = "Backup Unit" = memory card)
   right before makes this unambiguous.
2. **Two PSX BIOS event classes split 4/4**: `0xF4000001` (I/O-
   complete class) and `0xF0000011` (error/notification class).
3. **`pad_press_control` is a misnomer** -- it does memcard init,
   not pad-press handling. Likely a stale Kengo-derived rename.

**Refined name proposal**: `memcard_event_pool_close_80037774`
(replaces the over-generic `game_event_shutdown_*`). Promoted to
**high** confidence -- the OpenEvent owner identifies the 8-event
pool unambiguously.
