## Evidence

### Closes 8 BIOS events inside a critical section (rank=high)

The function takes the canonical "shutdown / unregister" shape:

1. `EnterCriticalSection()` -- prevent interrupts from racing.
2. `bios_CloseEvent(handle)` x 8 -- release every event handle owned
   by this module.
3. `ExitCriticalSection()` -- restore interrupt state.
4. `func_8007A400()` -- a post-shutdown callback (presumably resets
   the per-module state that referred to the now-closed events).

The pattern is unambiguous: this is the **shutdown / cleanup** half of
a per-module BIOS-event lifecycle. There is no other valid
interpretation of "close 8 events under a critical section" -- this
is the inverse of an OpenEvent chain elsewhere.

### The 8 event handles are *the same set* used by the partner waiter (rank=medium)

`func_80037964` (separately documented in
[func_80037964.md](func_80037964.md), proposed
`game_event_wait_any_80037964`) tests 4 of these handles
(`D_800A3838`, `D_800A383C`, `D_800A3848`, `D_800A3850`) in a
`bios_TestEvent` loop. The other 4 are presumably tested elsewhere
in the same module.

So the event handles `D_800A37DC` / `..F0` / `..FC` / `..3800` /
`..3838` / `..383C` / `..3848` / `..3850` are a coherent group --
the per-module event pool. This function unwinds all of them, which
is by definition the module's shutdown hook.

## Why this name

* `game_event_shutdown_<addr>` -- captures the "shutdown" role and
  the BIOS-event subsystem.
* `<addr>` suffix is the convention for an `_addr_` qualifier when
  the exact module isn't yet identified -- there are likely several
  per-module event shutdown helpers in BB2.
* Confidence **high** because the pattern (EnterCS + N CloseEvents +
  ExitCS + post-shutdown call) is a forced read; no plausible
  alternative interpretation.
