## Body shape

```c
s32 func_80080258(s32 a0, s32 a1, s32 a2) {
    register s32 result asm("s7");
    s32 count = 3;          // retry count
    s32 idx = a0 & 0xFF;
    s32 saved = g_cd_callback_a;
    s32 *elem = &g_cd_sector_buf[idx];

loop:
    g_cd_callback_a = 0;
    if (idx != 1 && (g_cd_mode & 0x10)) {
        tslTm2LoadImage(1, 0, 0, 0);     // first-pass reset / clear
    }
    if (a1 != 0 && (*elem) != 0) {
        if (tslTm2LoadImage(2, a1, a2, 0) != 0) {
            goto next;
        }
    }
    g_cd_callback_a = saved;
    if (tslTm2LoadImage(a0 & 0xFF, a1, a2, 0) == 0) {
        goto done;
    }
next:
    count--;
    if (count != -1) goto loop;
    g_cd_callback_a = saved;
    result = -1;
done:
    return result + 1;
}
```

## Evidence

### Wraps `tslTm2LoadImage` with retry (rank=high)

The function makes three (up to) calls to the named PsyQ-derived
helper `tslTm2LoadImage` per loop iteration:

* `tslTm2LoadImage(1, 0, 0, 0)` -- mode-1 reset (when `idx != 1`
  and `g_cd_mode & 0x10`).
* `tslTm2LoadImage(2, a1, a2, 0)` -- mode-2 secondary upload (only
  if the sector-buf slot is non-zero).
* `tslTm2LoadImage(a0 & 0xFF, a1, a2, 0)` -- the primary load with
  the caller-supplied operation code.

The outer `count = 3; ... count != -1` is a classic "try this 3 times
then bail" retry loop. The function returns `0` on success and `-1`
on failure (the `result + 1` pattern at exit).

### Manages `g_cd_callback_a` save/restore (rank=high)

The function saves the global `g_cd_callback_a` callback pointer at
entry, sets it to 0 for the duration of the load, and restores it
before each exit. This is the canonical "suppress callback while
issuing a synchronous CD read" pattern -- it prevents the CD
finish-callback from firing recursively while the function is
actively waiting for the load to finish.

### Wrapped specifically around tslTm2LoadImage (rank=medium)

Naming follows the most-specific PsyQ helper actually wrapped:
`tslTm2LoadImage` (likely "TIM-2D Load Image" in BB2's PSX SDK
fork). The function is essentially "retry the TIM-2D image load
with callback suppression and a hard cap of 3 attempts."

## Why this name

* `tim2_load_with_retry_<addr>` reflects:
  - `tim2` -- matches the underlying PsyQ helper `tslTm2LoadImage`
    (Tim-2 = the BB2 fork name for TIM data manager)
  - `load_with_retry` -- the retry-loop wrapper role
  - Address suffix preserved because this is one of several similar
    retry wrappers in `system.c` (other addresses in 0x800802x..
    region) and the exact "what differentiates this one from the
    others" is still TBD.

## Caller-side audit (2026-05-17 reinforcement) -- refines the proposed name

5 call lines show arg0 is NOT always a "TIM-related" operation -- it's
a generic command code:

* `func_80080258(2, index + cam_base, 0)` (code6cac_b2.c:629, retry
  block) -- mode 2 with a cam-table address.
* `func_80080258(2, (s32)buf2_ptr, 0)` (code6cac_b2.c:646) -- mode 2.
* `func_80080258(9, 0, 0)` (system.c:1001) -- mode 9, **arg1=0** (no
  data buffer at all).
* `func_80080258(2, temp_s0, 0)` (system.c:1003) -- mode 2.
* `func_80080258(0xE, (s32)&sp10, 0)` (system.c:1016) -- mode 0xE
  (14), with a stack-allocated data buffer.

Mode codes observed in callers: **{2, 9, 0xE}**. Mode 9 passes
arg1=0 (no data) -- inconsistent with "TIM image load", which would
require a TIM buffer. So the function is NOT specifically a
TIM-loader -- it's a generic `tslTm2LoadImage` command wrapper that
dispatches on the operation code (mode 1 = init, mode 2 = upload,
mode 9 = no-arg op like flush-cache, mode 0xE = clear-display-mask
or similar).

The PsyQ `tslTm2LoadImage` API takes its operation code as the first
arg precisely because it's not pure-TIM-load -- it's the "TIM-2D
combined operation dispatch" entry point. Our wrapper just adds
retry semantics.

**Refined name:** `tslTm2_command_with_retry_80080258` or
`tslTm2_call_with_retry_80080258` -- captures "this is a retry
wrapper around the generic tsl-Tm2 dispatcher, not specifically a
TIM-load helper".
