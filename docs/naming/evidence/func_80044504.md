## Evidence

### Initialises 5 scratchpad slots before per-frame compute (rank=high)

The PSX scratchpad lives at `0x1F800000..0x1F8003FF` (1 KB of fast
SRAM). This function writes 5 different scratchpad addresses:

| Addr           | Value source                                                   |
|----------------|----------------------------------------------------------------|
| `0x1F80000C`   | `a0` -- caller-supplied integer                                 |
| `0x1F800010`   | `func_8003E2C8()` if `g_game_timer & 8`, else `0x7FFFFFFF`     |
| `0x1F800014`   | `-1` if `D_800A36AC & 1`, else `0`                              |
| `0x1F800018`   | `0xBE` / `0x182` / `0xBE` depending on player count + pause     |
| `0x1F80001C`   | `&D_80095328` -- a fixed pointer                                |

5 scratchpad words are the per-frame compute kernel's inputs --
chosen because scratchpad reads/writes are much faster than DRAM
(no cache misses). This is the **frame-compute environment setup**
that runs before some hot loop that consumes the scratchpad values.

### Calls established helpers (rank=medium)

`camera_InitMatrix()`, `game_GetPlayerCount()`, `game_GetPause()`,
`game_SetPause()`, `calc_fc_frame_8007EC5C` are all already-named.
The structure (init matrix, set scratchpad inputs, call
`func_8004A4E0()` -- presumably the hot loop) matches the
"per-frame init + dispatch hot loop" pattern of PSX game-loop
helpers.

## Why this name

* `frame_context_init_scratchpad_<addr>` -- captures (a) the per-
  frame init role, (b) the scratchpad-as-fast-input pattern, (c)
  the "context init" (sets up everything needed before the hot
  compute).
* Confidence **medium-high** because the scratchpad-write pattern
  is unambiguous, the helper-call sequence anchors the per-frame
  init role, and `game_SetPause(1)` + `func_8004A4E0()` at the end
  matches "set up the compute environment then dispatch".
