## Evidence

### Inverse of the prim_buffer_open_slot pair (rank=high)

This function is the **release** half of the slot-pool lifecycle
managed by [`func_80044010`](func_80044010.md) (proposed
`prim_buffer_open_slot_*`) and [`func_80044100`](func_80044100.md)
(proposed `prim_buffer_relocate_slot_*`). It performs the exact
inverse of the open operation:

| Step          | Open (`func_80044010`)              | Release (this function)             |
|---------------|-------------------------------------|-------------------------------------|
| Header bit    | Sets `0x8000` (in-use marker)       | Clears `0x8000` (frees the slot)    |
| Header count  | Stores from passed argument         | Reads, decrements by 1, writes back |
| Payload walk  | Forward header-fixup (relocate +)   | Subtracts header_ptr (relocate -)   |
| Touch arrays  | Writes `D_80103608[a1]`, `D_80103658[a1]` | Reads `D_80103608[a0]`        |

The "un-relocate" step (`*v1 -= (s32)a6;` where `a6 = v1 - 1` =
header pointer) is the inverse of the open's header-fixup pass --
it converts each entry from "absolute address" back to "offset
relative to header" so the payload can be safely moved or freed
again.

This is **unambiguously the matching release / close-slot helper**
for the open/relocate pair. The slot pool is a 3-function lifecycle
(register/relocate/release) with these three address-clustered
functions.

### Register-asm pinning (rank=info)

The body uses `register s32 *v1 asm("v1");` and `register s32 a4
asm("a0");` -- explicit register hints. These are the project's
convention for matching a specific GCC 2.7.2 register-allocation
pattern that hand-decompilation flagged as the only one producing
byte-matching codegen. Not naming-relevant but worth noting.
