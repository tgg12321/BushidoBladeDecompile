## Evidence

### Stores into `g_player_ptrs[idx]` (rank=high)

The function takes a player slot index `a0`, calls a creator
`func_80045878(a0)` to obtain a Rob pointer, and stores that pointer
into `g_player_ptrs[idx]` -- the named global player-pointer table.
That alone identifies the function as a per-player setup hook called
during fight initialisation.

### Calls a chain of already-named `rob_*` initialisers (rank=high)

After the creator call, the function invokes:

* `AllocRobRmd(ptr)` -- allocate the per-player Rob "RMD" (likely
  resource/model data) block.
* `rob_life_ctrl(ptr)` -- initialise lifebar / HP controller.
* `rob_calc_2d_position(ptr)` -- compute the 2D screen-space
  position.
* `func_8003F824(ptr, 1)` -- a sibling helper, 2-arg pattern
  (probably an init flag).
* `FadeOut_8003FFC4(ptr)` -- per-player fade controller setup.
* `func_80040CB8(ptr)` -- another per-Rob setup helper.
* `func_8003E120()` -- a global side-effect (no arg) -- likely
  "notify global state that a new player was registered".

This chain is the **canonical per-player initialisation sequence**
expressed through the existing `rob_*` named helpers.

### `do { ... } while (0)` wrapper (rank=info)

The `do { ... } while (0)` around three of the calls is a maspsx /
GCC 2.7.2 idiom for forcing inline-block scheduling; the body
semantically just runs the three calls sequentially. It does not
imply any iteration or retry.

## Why this name

* `player_rob_Init` follows the project's existing naming convention:
  `motion_Open`, `motion_SetMotion`, `cdrom_SetCallbackB`,
  `disp_SetFramebufferMode` -- "subsystem_VerbNoun" with the verb in
  CamelCase. The Rob is the per-player struct; `Init` is the action.
* The function fits the slot of a public "create a player at slot N"
  entry point. The existing companions `AllocRobRmd`, `rob_life_ctrl`,
  `rob_calc_2d_position` are all per-player init helpers; the parent
  that orchestrates them is the *one* that has been unnamed -- a
  high-impact rename to anchor the rest of the family.
