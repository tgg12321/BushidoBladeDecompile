## Evidence

### Pairs with `func_80044010` on the same two arrays (rank=high)

Operates on the same `D_80103608[]` and `D_80103658[]` arrays that
`func_80044010` populates (see that function's evidence file). The
pair is the entire reader/writer set for this structure -- no other
function in `src/` touches either array.

`a0` is the slot index; `a1` is a byte-count delta.

Action:
1. Get the slot's current payload pointer and entry count.
2. Advance the stored payload pointer by `a1 / 4` entries (= `a1`
   bytes, since each entry is a `s32`).
3. Walk the original payload (count - 1 iterations) and add `a1`
   bytes to each entry value.

### Pattern matches "relocate an offset-list" (rank=medium)

When `*entry` values are byte offsets into a linear buffer, and the
buffer base shifts by `a1` bytes, every offset stored in the entry
list needs to be adjusted by the same `+a1`. That is exactly what the
loop does.

This pattern is characteristic of:

* PSX OT (ordering table) relocation when the table moves in memory.
* PSX prim-list back-patching when the prim buffer is reallocated.
* Any compact "offset table" structure that supports buffer migration.

The `prim_buffer` naming follows from the assumption that this pairs
with the prim-buffer slot opener in `func_80044010`.

### Caller-shared idiom (rank=low)

Several callers pass `a1` as a fixed compile-time constant (slot
sizes 0x40, 0x80, 0x100 etc.). This is consistent with "move the
write-pointer forward by a fixed slot size and patch the existing
entries to point at the new region."
