Every case ends up making one or more calls into the `func_8006xxxx`

is a big event-dispatch table that maps an `arg1` event code (1..50)
to a per-motion-slot offset lookup or to a specialised draw helper.

## Evidence

### Caller-cluster: 19 callers all in the motion subsystem (rank=medium)

* 17 callers in `src/code6cac_b.c` (same file -- internal helper)
* 2 callers in `src/code6cac.c`

`code6cac_b.c` is the per-frame motion / state-machine cluster that
manages `D_800A38DC` (player-state byte, values 1..5 visible in the
switch prologue) and the per-motion-slot data pool at offsets like
`s0 + 0x22`, `s0 + 0x35`, `s0 + 0x3E` etc. The function is wedged
right at the centre of that cluster.

### Functional signature: 4-arg dispatcher with `arg1` switch (rank=medium)

The 4-arg shape `(s32, s32, u8*, s16*)` is recognisably:

* `arg0` -- a small player/character slot index (multiplied by 40 in
  the s0 calc; matches the 40-byte per-motion-slot record stride seen
  elsewhere in code6cac_b.c).
* `arg1` -- an event / animation-frame code (1..50, used as the switch
  selector).
* `arg2` -- a `u8*` to a per-character motion buffer; passed verbatim
  to every per-case helper.
* `arg3` -- a `s16*` to a small auxiliary buffer (frame index, hit
  flag); read by the case-1, case-4, case-11, case-14 paths.

This is the classic shape of a "process motion event N for character M
using buffers B and C" dispatcher. The 30+ cases are individual
per-event handlers; the s0/s4 prologue computes the slot offset for
the entire family.

### `func_800395B4` is called first in every path (rank=info)

Looks like a per-call hook (maybe a frame-counter increment or audit
write) that fires before the dispatch. This is consistent with a
shared event-dispatch entry point.

## Why this name

* The function is a centralised dispatcher inside the motion subsystem;
  `motion_dispatch_event_handler_<addr>` captures (a) the subsystem,
  (b) the dispatcher role, and (c) the address suffix that this
  precise dispatcher's exact name is still a best-guess (multiple
  similar dispatchers may live in the file).
* The address-suffix is recommended because the analyzer flags
  ambiguous matches by suffixing -- this is a `medium`-rank proposal
  precisely *because* the exact slug ("event-handler" vs
  "frame-stage-handler" vs "anim-tick-dispatch") is not pinned down by
  evidence alone.

## Reviewer alternatives

If a future trace through `code6cac_b.c` reveals that `arg1` values are
animation-stage codes from a specific table, a more specific name like
`motion_stage_dispatch_<addr>` or `motion_run_stage_<addr>` would be
better. The medium-confidence rank reflects that uncertainty.
