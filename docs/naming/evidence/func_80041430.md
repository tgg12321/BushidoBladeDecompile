## Evidence

### Relocation pattern matching `func_80044100` (rank=high)

Same shape as the TPAGE-slot relocator
[`func_80044100`](func_80044100.md) (proposed
`prim_buffer_relocate_slot_*`) -- "advance the base pointer by N
bytes, then walk every embedded sub-pointer in the payload and
patch them by +N":

* `g_player_ptrs[a0]` is the player-N pointer slot. Reading
  `*base + a1` and storing back is the "shift by a1 bytes" step.
* The three `save_vc_ctrl(a1, &s0->field, kind)` calls likely patch
  sub-pointers stored at fixed offsets (`+0x2C`, `+0x8B4`, `+0x10D4`)
  inside the player struct.
* The two iteration loops walk chains of pointer-fields (offset
  `+0x112C` with 0x68 stride; offset `+0x1A34` with 4-byte stride
  for 0x14 entries) and patch each.
* Final `func_80040A78((s32)s0)` is the "post-relocate hook".

This is the **per-player memory-region relocator**, used when
shrinking / expanding / moving the player's data block in the
upper-RAM region (the same region documented in
[`g_motion_data_base_ptr`](../data_evidence/D_800A3860.md)).

### Pairs with `player_rob_Init` (rank=medium)

[`func_80040510`](func_80040510.md) (proposed `player_rob_Init`)
sets up the player at `g_player_ptrs[slot]` initially. This function
relocates a previously-initialised player by `a1` bytes -- the
"shift the player's heap region after a different player was
unloaded" path.

## Why this name

* `player_rob_relocate_<addr>` -- matches the `player_rob_*` family
  established by `player_rob_Init`. Captures (a) the player /
  rob role, (b) the relocation semantics.
* Confidence **medium-high** because:
  1. The relocation pattern (advance base + patch embedded ptrs) is
     mathematically unambiguous.
  2. The `g_player_ptrs[a0]` access locks it into the player subsystem.
  3. The exact role of the embedded patch-tables at `+0x2C`,
     `+0x8B4`, `+0x10D4`, `+0x112C`, `+0x1A34` would refine the
     name (perhaps `player_rob_grow_<addr>` if it's always growing,
     or `player_rob_compact_<addr>` if it's shrinking).
