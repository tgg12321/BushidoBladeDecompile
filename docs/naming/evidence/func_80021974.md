## Evidence

### The 1100-byte stride is per-character state (rank=medium)

`a0 * 1100` reads from `D_80101F12` and `D_80101F4C` -- two fields
inside the same per-character struct whose row size is 1100
(`0x44C`) bytes. 1100 is large enough to hold a full per-character
state record (motion state, health, position vectors, etc.) and
matches the per-character storage stride elsewhere in code6cac.c.

`a0` is therefore a small character/player index (0..7 typically).

### Two interleaved table indexings (rank=medium)

The function chains two lookups:

1. `v1` -- read from the per-character record at +0x... (D_80101F12).
   Then `a0_2 = v1*4` is used as a byte offset into `D_800A3860`
   (table of 4-byte entries, indexed by `v1`).
2. `v0` -- read from the per-character record at +0x... (D_80101F4C).
   Then `idx = *(u16 *)(base + v0*2 + 0x4E)` looks up a 16-bit index
   from the previously-fetched base + (v0 entry * 2) + 0x4E.
3. Finally `tbl = *(s32 *)(D_801027B0 + v1_2*4)` (a parallel table
   indexed by `v1`, but 5x stride relative to a0_2 -- `(a0_2 + v1)*4`
   = `v1 * 5 * 4` = `v1 * 20`), and the return is `tbl + idx * 2`.

The output `tbl + idx * 2` is a pointer-sized value pointing to a
16-bit-element table. This shape -- "given a character index, look up
a base pointer in one table and an entry-index in another, return
the address of a 16-bit element" -- is the canonical "get current
keyframe pointer from a motion table" pattern in PSX animation
runtimes.

### No callers cluster (rank=low)

8 callers across code6cac.c and other files -- broad usage consistent
with a per-frame helper called by many paths in the motion/character
runtime.
