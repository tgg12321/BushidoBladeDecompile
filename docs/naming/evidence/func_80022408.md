## Evidence

### Pure 2D-distance scanner over 4 records (rank=high)

The function reads from `stage_GetDataPtr()` (already-named), offset
by `D_800A36A4 * 24` halfwords (= `D_800A36A4 * 48` bytes). Each
record is 6 s16 fields (12 bytes), so the function inspects 4 such
records.

For each record it computes:

* `midpoint_x = (record.s16[0] + record.s16[3]) / 2`
* `midpoint_z = (record.s16[2] + record.s16[5]) / 2`
* `dist = (midpoint_x - arg0.x)^2 + (midpoint_z - arg0.z)^2`

and returns the index `i` (0..3) of the record with the smallest
distance. Classic "find nearest of N corners" pattern.

### The 4-record layout fits a quad / corner set (rank=medium)

The "6 s16 fields per record" pattern with `[0]` and `[3]` as a
linked pair (the midpoint formula `(s0 + s3) / 2`) suggests each
record stores two endpoints:

```
record = { s16 ax, s16 ay, s16 az, s16 bx, s16 by, s16 bz; }
```

i.e., a line segment with start and end vertex. The midpoint is the
segment's centre. With 4 such segments, the "stage data" being
queried is likely a stage boundary or four-corner enclosure (the
fighting area's 4 wall midpoints).

`arg0[0]` and `arg0[2]` are read as s32 (the function declared
`s32 *arg0`), but the comparison uses only the low 16 bits implicit
in the s16 subtraction. So `arg0` is a `(s32 x, s32 _, s32 z, ...)`
struct -- the standard PSX VECTOR pattern where coordinates are
stored as 32-bit values but the comparison drops the high bits.
