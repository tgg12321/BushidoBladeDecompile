## Evidence

### Pairs with `func_80044100` on the same two arrays (rank=high)

The two arrays `D_80103608[]` and `D_80103658[]` are touched by exactly
two functions in the entire src tree: this one (`func_80044010`, the
"writer/opener") and the sibling `func_80044100` (the "advancer/relocator"
-- see its evidence file). Together they implement a simple paired-slot
data structure:

* `D_80103608[a1]` -- pointer to the slot's payload
* `D_80103658[a1]` -- count of payload entries (15-bit; the 16th bit
  is the "in use" / "open" marker on the header)

`func_80044010(header, slot)` opens a slot by marking the header word
with bit 0x8000 ("in use") and storing the payload pointer + count
into the two parallel arrays. Then it walks the payload to perform a
header-fixup pass.

### `a1` is a small slot index (rank=medium)

Callers pass `a1` as a constant small integer (0, 1, 2, ...). This is
consistent with a fixed-size pool of named slots -- the index is
chosen by the caller per-purpose, not allocated dynamically.

### Cross-file caller spread suggests a shared primitive (rank=low)

9 callers across 4 files (sound.c, text1a.c, text1a_c.c, text1a_c2.c)
indicates a general-purpose helper used by multiple subsystems.

## Why this name

The function "opens" a slot by setting the header's `0x8000` bit,
records the slot's payload pointer + length in the parallel
`D_80103608` / `D_80103658` arrays, and then iterates the payload
performing a fixup pass. The `prim_buffer_open_slot` slug captures:

* "prim_buffer" -- the BB2 PSX primitive-list buffer pattern of header
  word + payload entries (consistent with the 0x8000-bit marking which
  matches the PSX OT/prim-list "last entry" / "terminator" convention
  in some BB2 helpers).
* "open_slot" -- the action of marking the slot as in-use and recording
  its location.

Address suffix retained at medium confidence because (a) the exact
semantics of the post-open fixup loop are not yet decoded, and (b)
without confirming the payload is indeed a PSX prim-list (vs. a
generic record array), the "prim_buffer" prefix is a strong guess
rather than a certainty.

## Caller-side audit (2026-05-17 reinforcement) -- refines the proposed name

8 call lines clarify the data being registered:

* `func_80044010(texA, ((s16 *)a0)[10])` (text1a.c:259) -- `texA` is
  declared as a `s32 *` derived from `(s32)rmd + (((u32)rmd[3] >> 2) << 2)`
  -- a **texture page** pointer derived from an RMD model.
* `func_80044010(texB, ((s16 *)a0)[11])` (text1a.c:260) -- same
  shape, different texture page.
* `func_80044010(s0, 9)` (sound.c) -- sound-related buffer registered
  to slot 9.
* `func_80044010(PTR_OFF(s0, ALIGN4(s0[5])), 8)` (text1a_c2.c) --
  aligned offset into an RMD packed structure.
* `func_80044010((s32)s6, 7)` (text1a_c2.c:162)
* `func_80044010(a0, 7)` (text1a_c2.c:201)
* `func_80044010(s2, 7)` (text1a_c2.c:234)

The pattern is **strongly TPAGE-shaped** -- arg0 is a 4-byte-aligned
pointer to model data, arg1 is a small slot number in `{7, 8, 9, 10,
11}`. This corresponds to PSX VRAM TPAGE / CLUT slot management:
slots 7..11 are likely the per-character / per-mode texture-page
allocations.

The Kengo match (`HiraRmdAddTbpOfst` -- "Hiragana RMD add TBP
offset") is `size-only-ambiguous` with 3 candidates and
opseq_ratio=0.37 -- weak, but the Kengo name's prefix `RmdAddTbp`
(= "RMD AddTextureBP") is *consistent* with the caller-side
interpretation. This convergence promotes the slug from
"prim_buffer_open_slot" to a more accurate **`rmd_AddTbp_80044010`**
or **`tpage_register_slot_80044010`**.
