# D_80103608 -- naming evidence

- Address: `0x80103608`
- Access width: 4 bytes (s32 pointer) per element
- Section: BSS / writable RAM
- Touchers: **12** functions (2 src files, 10 asm files):
  - Manager pair: `func_80044010` (registers), `func_80044100` (relocates)
  - Consumers: `calc_loc_mat_fw_8004A940`, `func_8003FA24`,
    `func_800433E4`, `func_80044098`, `videoDecCreate`, and 5 others
- Proposed name: `g_tpage_slot_ptr`
- Confidence: **medium-high**

## Manager pair

The two functions that *manage* this table are documented in detail
in their evidence files:

* [func_80044010.md](../evidence/func_80044010.md) -- registers a
  slot. After my caller-side audit:
  `func_80044010(*ptr, slot_idx)` -- stores `*ptr` into
  `D_80103608[slot_idx]` and the corresponding payload-count into
  the parallel array [`D_80103658`](D_80103658.md).
* [func_80044100.md](../evidence/func_80044100.md) -- relocates a
  slot. `func_80044100(slot_idx, byte_delta)` -- advances the stored
  pointer in `D_80103608[slot_idx]` and patches every entry in the
  pointed-to payload by `+byte_delta`.

The pointer-and-count pair is exactly the runtime representation of a
TPAGE / CLUT slot pool:

```
D_80103608[N]  = pointer to slot N's payload start (post-header)
D_80103658[N]  = count of payload entries in slot N (15 bits, the
                 16th bit lives on the header word)
```

## Slot indices observed

Across caller-side audits of the manager pair and the 10 consumer
asm functions, the slot indices touched are `{6, 7, 8, 9, 10, 11}`
plus one global-pointer-stored index (`D_800A9CF8`). The table is
therefore at least 12 slots wide (probably 16 or 32 -- typical PSX
TPAGE pool sizes). Each slot is dedicated to a specific
texture-page resource type (per-character textures, common UI
sprites, font glyphs, etc. -- the exact slot-to-resource mapping is
documented per-consumer in the manager evidence files).

## What the consumers do

A spot check of 3 of the 10 asm consumers:

* `videoDecCreate` -- reads `D_80103608[N]` and `D_80103658[N]` to
  iterate the payload entries of a video-decoder TPAGE slot.
* `func_80044098` -- a small helper that takes a slot index and
  resets the in-use bit on its header word. Sibling to the
  manager pair.
* `calc_loc_mat_fw_8004A940` -- reads `D_80103608[N]` as a base
  address for matrix data inside a TPAGE slot.

All consumers treat the value at `D_80103608[N]` as a pointer to
binary data (texture pages, CLUTs, font glyphs, model auxiliary
buffers) that the slot manages. No consumer interprets the value as
anything other than a pointer.

## Why this name

* `g_tpage_slot_ptr` -- captures (a) the array structure
  (one-pointer-per-slot), (b) the TPAGE/CLUT runtime role
  identified by the texA/texB caller-side evidence in the manager
  pair, and (c) the "slot" terminology that matches how callers
  refer to slot indices.
* Confidence **medium-high** because:
  1. The manager pair is unambiguously identified.
  2. The 15-byte stride pattern + parallel-array pattern matches
     the PSX texture-page-pool runtime exactly.
  3. The "TPAGE" wording is justified by the texA/texB args passed
     by the registers (texture page pointers from RMD models).
* Not full **high** because: alternative interpretations like
  "generic resource slot pool" or "scratch buffer registry" are
  also consistent with the access pattern. The TPAGE-specific
  reading is the most likely but not the only option.
