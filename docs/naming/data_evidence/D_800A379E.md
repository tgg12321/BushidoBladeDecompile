# D_800A379E -- naming evidence

- Address: `0x800A379E`
- Access width: 2 bytes (s16) -- single scalar
- Section: BSS / writable RAM
- Touchers: **7+** functions (12 asm hits, 13 src hits in
  `code6cac_c_mid.c`)
- Already in `data_symbols_quick_wins.md` as `stbl_800A379E`
  (mechanical-width guess) -- this evidence file upgrades that to a
  semantically-informed name.
- Proposed name: `g_motion_state_code`
- Confidence: **medium-high**

## Sole reader-and-return of interest

    return D_800A379E;         // current motion-state code

The callers (3 of them, all in `code6cac_c_mid.c` motion subsystem)
use the return value as a state-machine selector via `switch (v0 - 4)`.

## Writers

Multiple `motion_*` helpers write D_800A379E (`sh` to the address):

* `func_80038658` -- the per-frame state-update helper called by
  `game_state_advance` after `pad_FuncAnalog`. Writes 3 times in its
  body (multiple branches of the state machine).
* `func_8003879C`, `func_800387C0`, `func_800387E8` (already
  proposed as `motion_SetMotion_helper_*` in the existing analysis)
  -- each writes the state code in motion-state transition paths.
* Several other motion-cluster functions write it from inside their
  per-stage handlers.

The dominant pattern is `sh` (store halfword) -- consistent with the
2-byte access width that `data_symbols_quick_wins.md` flagged.

## Why this name

* `g_motion_state_code` -- captures (a) the motion subsystem all
  writers live in, (b) the "state code" role (a small enum-like s16
  that callers `switch` on), and (c) the fact that this is the
  *return value* of the state-advance entry point.
* Confidence **medium-high** because:
  1. The return-value role is unambiguous (caller uses it for
     `switch`).
  2. All writers cluster in the motion subsystem.
  3. The 2-byte access width matches the mechanical analysis.
* Not full **high** because: the exact value-to-state mapping isn't
  documented. A reviewer who can read the `switch` arms in the 3
  callers of `game_state_advance` could enumerate the state codes
  and possibly refine the name (e.g., `g_motion_phase` if the codes
  are linear phases vs. `g_motion_command` if they're command codes).
