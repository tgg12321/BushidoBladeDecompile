# D_801027B0 -- naming evidence

- Address: `0x801027B0`
- Access width: 4 bytes (s32 pointer) per element
- Access stride: 20 bytes per record (5 s32 fields)
- Section: BSS / writable RAM
- Touchers: **1** function -- the only reads are inside
  `func_80021974` (= `char_get_motion_keyframe_table_entry_*`); both
  reads of the table appear in two siblings of the same shape (lines
  2509, 2520)
- Proposed name: `g_motion_keyframe_table_ptr`
- Confidence: **medium**

## Sole reader

```c
// src/code6cac.c -- func_80021974 (char_get_motion_keyframe_table_entry)
s32 func_80021974(s32 a0) {
    s32 offset = a0 * 1100;
    s16 v1 = *(s16 *)((u8 *)&D_80101F12 + offset);
    s16 v0 = *(s16 *)((u8 *)&D_80101F4C + offset);
    s32 a0_2 = v1 * 4;
    s32 v1_2 = (a0_2 + v1) * 4;       // (v1*4 + v1) * 4 = v1 * 20
    s32 base = *(s32 *)((u8 *)&D_800A3860 + a0_2);
    u16 idx  = *(u16 *)(base + v0 * 2 + 0x4E);
    s32 tbl  = *(s32 *)((u8 *)&D_801027B0 + v1_2);    // <-- this
    return tbl + idx * 2;
}
```

The expression `(u8 *)&D_801027B0 + v1_2` where `v1_2 = v1 * 20`
means D_801027B0 is the base of an array of 20-byte records,
indexed by `v1` (a value from the per-character struct at offset
0x12).

We READ a single `s32` from each indexed record (no field offset
inside the 20-byte record is used in this function alone). So either:

* The 20-byte record contains a single s32 pointer at offset 0 plus
  16 bytes of additional metadata the caller doesn't touch, OR
* `v1_2 = v1 * 20` is the byte offset into a flat array of `s32`
  that happens to step by 5 entries at a time (i.e., 5 motion-table
  pointers per character, with `v1` selecting the character).

The second interpretation is the more likely:
**D_801027B0[char_idx]** = pointer to that character's
motion-keyframe table, where each character has 5 such tables
(perhaps "idle / move / attack / damage / death" or similar). The
`+0x4E` offset deep-into-base earlier in the function reads a
*current keyframe index*, and `tbl + idx * 2` gives a `u16 *` to
that keyframe entry.

## Why this name

* `g_motion_keyframe_table_ptr` -- captures (a) the array-of-pointers
  shape, (b) the motion-system context, and (c) the keyframe-table
  role identified by the return-value computation
  `tbl + idx * 2` (where idx is a small u16 read from offset 0x4E
  of the per-character struct).
* Confidence **medium** because:
  1. Only one function reads it -- limited view of the data's role.
  2. The 20-byte stride is unambiguous from the asm/C, but its exact
     interpretation (5-pointer record vs. 1-pointer + 16-byte padding
     vs. 5-stride-with-2-keyframes-each) needs more evidence.
  3. The per-character indexing matches the partner data symbol
     `D_800A3860` (also per-character keyframe-related).

## Reviewer alternatives

* `g_motion_segment_tbl_ptr` -- if the "keyframe table" reading is
  too narrow.
* `g_motion_stage_table_ptr` -- if `v1` is more like a state-stage
  index than a per-character index.

## Confidence-raising next steps

* Find a writer of D_801027B0 (none observed in code6cac.c -- it
  might be set from outside this file, perhaps by the motion data
  loader).
* Identify the OTHER 4 s32 fields in the 20-byte record by looking
  for code that reads `D_801027B0[i].field_4`, `field_8`, etc.
* If no other fields are ever read, the 20-byte stride is actually
  a 5-element-per-character pointer array -- which would refine the
  name to `g_per_char_motion_table[][5]` or similar.
