# D_80103658 -- naming evidence

- Address: `0x80103658`
- Access width: 4 bytes (s32 count) per element
- Section: BSS / writable RAM
- Touchers: **3** functions:
  - Manager pair: `func_80044010` (registers), `func_80044100` (uses
    for relocation loop bound)
  - `videoDecCreate` -- single consumer that uses slot 9's count
- Proposed name: `g_tpage_slot_count`
- Confidence: **medium-high**

## Role

Parallel array to [D_80103608](D_80103608.md). For each slot `N`:

```
D_80103608[N]  = pointer to slot's payload data
D_80103658[N]  = number of (s32) entries in the payload
```

The count is stored as a 15-bit value (the 16th bit `0x8000` lives on
the original header word as an "in-use" marker, set by
[`func_80044010`](../evidence/func_80044010.md) before storing the
count). The manager pair uses the count to bound the relocation loop:

```c
// src/text1a_c.c -- func_80044100 body (partial)
s32 *ptr = D_80103608[a0];
s32 count = D_80103658[a0];
D_80103608[a0] = (ptr = ptr + (a1 / 4));   // advance the pointer
count--;
do {                                        // walk the (count-1) entries
    *ptr += a1;
    ptr++;
    count--;
} while (count != -1);
```

The do-while iterates `count` times because of the `count--` before
the loop and the `count != -1` exit. So `D_80103658[N]` is literally
the number of `s32` entries the manager iterates.

## Why this name

* `g_tpage_slot_count` -- "count" because the value is a count of
  entries (a number), not a pointer.
* Confidence **medium-high** -- same reasoning as the partner
  [D_80103608](D_80103608.md). The "TPAGE" qualifier inherits from
  the partner's caller-side evidence.

## Rename in lockstep

If [D_80103608](D_80103608.md) is renamed to a different slug, this
should match -- e.g., `g_rsrc_slot_ptr` + `g_rsrc_slot_count`. The
two arrays are an inseparable pair; renaming them inconsistently
would obscure the relationship.
