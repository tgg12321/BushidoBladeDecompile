# D_800F1848 -- naming evidence

- Address: `0x800F1848`
- Access width: 4 bytes (u32) -- single scalar
- Section: BSS / writable RAM
- Touchers: **2** functions (1 src file, 2 asm files):
  - `func_80079154` (= `bb2_rand`) -- 4 references (read + write)
  - `func_80079184` -- 2 references (read)
- Proposed name: `g_rand_state`
- Confidence: **high**

## Body of the only writer

```c
// src/text1b_b.c -- func_80079154 (proposed: bb2_rand)
s32 func_80079154(void) {
    D_800F1848 = D_800F1848 * 0x41C64E6D + 0x3039;
    return (D_800F1848 >> 16) & 0x7FFF;
}
```

This is the entire write path. `D_800F1848` is the seed/state variable
of the canonical BSD libc `rand()` linear congruential generator
(see [func_80079154.md](../evidence/func_80079154.md) for the constants
analysis). There are no other writers anywhere in the binary -- the
LCG state is owned exclusively by the rand helper, just like
`static unsigned long next` in the original libc implementation.

## Reads outside the writer

`func_80079184` (an immediate-neighbour function -- 0x30 bytes after
the writer) reads `D_800F1848`. The src form is:

```c
// src/text1b.c:13271
func_80079184(func_80079154());        // pass rand() result to a sibling
```

`func_80079184` itself is currently unnamed, but is probably a thin
wrapper that uses the most-recent rand value (perhaps a "use the
seeded value to pick an entry from a small table"). It does NOT
re-seed -- it only reads.

## Why this name

* `g_rand_state` -- the standard libc-source-code name for the variable
  that holds the LCG state between calls. Matches the convention used
  by glibc / BSD libc / PSX SDK and is immediately recognisable.
* Prefix `g_` matches the project's existing convention for named
  global RAM variables (`g_cd_callback_a`, `g_cd_mode`,
  `g_player_ptrs`, `g_file_dma_flag`, `g_str_*`).
* Confidence **high** because (a) the only writer is identified as
  `bb2_rand` with high confidence, (b) the LCG-state interpretation is
  the only one consistent with the read/write pattern (state is read,
  multiplied, added, stored back, then a derivative returned), and
  (c) no other function modifies it.

## Reviewer alternatives

* `g_rng_state` -- if the project prefers "rng" over "rand".
* `g_rand_seed` -- if the project prefers "seed" (slightly inaccurate
  since the value evolves; a true "seed" is the initial value, not
  the rolling state).
* `g_bb2_rand_state` -- if longer disambiguation is desired.
