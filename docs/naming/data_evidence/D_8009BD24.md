# D_8009BD24 -- naming evidence

- Address: `0x8009BD24`
- Section: BSS / writable RAM
- Access pattern: struct base address (most touchers load `&D_8009BD24`
  and iterate via pointer)
- Touchers: **6** asm functions, 2 src files (text1b.c, text1b_b.c)
- Proposed name: `g_disp_config`
- Confidence: **medium-high**

## Sole writer (initialiser)

`func_80077820` (= [`disp_load_config_from_buf_80077820`](../evidence/func_80077820.md))
is the only function that writes the *whole* struct -- it
loads `&D_8009BD24` into `$a1` and calls
`func_80068F70(src_buf, &D_8009BD24)`, which copies a config payload
from the caller-supplied source buffer (always `0x80118800`, the
upper-RAM staging area) into this struct:

```
asm/funcs/func_80077820.s:
  lui   $a1, %hi(D_8009BD24)
  addiu $a1, $a1, %lo(D_8009BD24)
  jal   func_80068F70                  ; memcpy-shaped helper
```

Per [the func_80077820 evidence](../evidence/func_80077820.md), this
is the "load display configuration from a staging buffer into the
in-RAM display-config struct" path.

## Readers

Five other asm functions touch the struct:

* `func_8005E54C` -- 2 references; reads from the struct (gpu setup
  helper region 0x8005Exxx).
* `func_80060414` -- `lbu %lo(D_8009BD24)($v0)` reads the first byte
  directly. Treats D_8009BD24 as a u8.
* `func_80077984` -- adjacent to the writer (within 0x164 bytes),
  presumably reads recently-loaded config.
* `func_80077A80` -- same neighbourhood.
* `saTan2GaugeInit_80077D00` -- already-named ("saTan" = `sa` =
  ScreenArea / "Tan" = tangent / "GaugeInit" = gauge initialiser),
  loads `&D_8009BD24` and iterates fields starting from there. This
  is a gauge / HUD-element initialiser that reads display config to
  position itself.

The mix of byte-level access (`lbu`) and address-of access
(`lui+addiu` pairs followed by `lw`/`lh` at small offsets) indicates
a STRUCT with multiple fields of different widths, not a single
scalar.

## Why this name

* `g_disp_config` -- captures (a) the "display" subsystem indicated
  by the writer's adjacent `disp_SetFramebufferMode(1,0,0,0)` call,
  (b) the "config" role indicated by the load-from-staging-buffer
  pattern, and (c) the multi-reader struct layout.
* Confidence **medium-high** because:
  1. The writer is unambiguously identified.
  2. The struct interpretation is forced by the mix of byte/word
     accesses at small offsets from the base.
  3. The display-subsystem context is established by the writer's
     `disp_SetFramebufferMode` neighbour.
* Not full **high** because: the exact field layout is undocumented,
  and a refined name like `g_disp_mode1_config` (specifically for
  mode-1 framebuffer state) might be more accurate if the writer is
  truly only used in mode-1 contexts.

## Confidence-raising next steps

* Document the struct field layout by reading what each of the 5
  reader functions extracts (each `lw`/`lh`/`lbu` at offset N from
  `&D_8009BD24` gives one field's width and role).
* Confirm the writer is only ever invoked from mode-1 paths, or find
  a different mode-2 writer that proves `g_disp_config` is the
  general struct (and `func_80077820` is just the "load and switch
  to mode 1" variant).

## Reviewer alternatives

* `g_disp_state` -- if "config" feels wrong (the struct could be
  per-frame mutable state, not setup config).
* `g_gpu_env` -- if it turns out to be a `DISPENV`/`DRAWENV`-shaped
  struct from the PsyQ libgpu API.
