# D_800A3860 -- naming evidence

- Address: `0x800A3860`
- Access width: 4 bytes (s32 pointer) per element
- Section: BSS / writable RAM
- Initialised at: `src/code6cac.c:2413` (`D_800A3860 = (s32)0x80148800;`)
- Touchers: **3+** functions in `src/code6cac.c` (one initialiser, two
  readers in the per-character motion lookup path; siblings
  `D_800A3830` and `D_800A3864` are initialised at adjacent lines)
- Proposed name: `g_motion_data_base_ptr`
- Confidence: **medium-high**

## Initialisation block

The variable is initialised next to two sibling pointer slots, all in
the same setup function around line 2412:

```c
// src/code6cac.c:2412
D_800A3830 = (s32)0x80120000;
D_800A3860 = (s32)0x80148800;       // <-- this symbol
D_800A3864 = (s32)0x80190800;
```

Note the addresses:

| Symbol      | Initial value | Region                                 |
|-------------|--------------:|----------------------------------------|
| D_800A3830  | `0x80120000`  | Upper RAM, ~1.13 MB into address space |
| D_800A3860  | `0x80148800`  | Upper RAM, ~1.28 MB into address space |
| D_800A3864  | `0x80190800`  | Extra-RAM region (BB2-special area)   |

The three pointers point into three distinct upper-RAM regions used
for streaming-loaded per-character motion / skeleton data
(D_800A3864 = `0x80190800` is the same address passed as the
"heap_base" arg by `player_rob_Init`'s call site -- see
[func_80040510.md](../evidence/func_80040510.md)). So the three are
likely **per-character data-region base pointers**: one slot per
loaded character (player 1, player 2, plus maybe a shared / NPC
slot).

## Reads as a pointer-table

`D_800A3860` is read as the base of a pointer ARRAY (the &-of pattern
plus byte-offset indexing):

```c
// src/code6cac.c:2491 -- read indexed by a struct field
s16 *v = (s16 *)(&D_800A3860)[arg0[0x4A / 2]];
arg0[0x86 / 2] = (s16)((a1 + 1) % v[0x14 / 2]);

// src/code6cac.c:2507, 2518 -- read indexed by per-character v1
s32 v1_2 = (a0_2 + v1) * 4;
s32 base = *(s32 *)((u8 *)&D_800A3860 + a0_2);    // a0_2 = v1 * 4
u16 idx = *(u16 *)(base + v0 * 2 + 0x4E);

// src/code6cac.c:2526, 2533 -- read indexed by struct field v0
s16 v0 = *(s16 *)((u8 *)&D_80101F12 + offset);
s32 base = *(s32 *)((u8 *)&D_800A3860 + v0 * 4);  // pointer fetch
u16 idx = *(u16 *)(base + 0x16);                   // read field inside
```

In every read, `D_800A3860` is treated as the BASE of an array of
`s32 *` pointers, indexed by a small integer (a character index, or
a motion-stage index). The fetched pointer is then dereferenced to
read packed per-character motion data (halfword fields at offsets
like `+0x14`, `+0x16`, `+0x18`, `+0x4E`).

## Companion: `func_80021974` uses this pointer table

[`func_80021974` (= `char_get_motion_keyframe_table_entry_*`)](../evidence/func_80021974.md)
performs the chained lookup:

```c
s32 base = *(s32 *)((u8 *)&D_800A3860 + a0_2);   // get base ptr for char
u16 idx = *(u16 *)(base + v0 * 2 + 0x4E);         // read keyframe index
s32 tbl = *(s32 *)((u8 *)&D_801027B0 + v1_2);     // get motion-table ptr
return tbl + idx * 2;                              // pointer to entry
```

This is the canonical "get the current motion keyframe for character
N" path: use `D_800A3860[char]` to find that character's data block,
read the current frame index out of that block, then use
`D_801027B0[stage]` to find the keyframe array, return the slot.

## Why this name

* `g_motion_data_base_ptr` -- captures (a) the array-of-pointers
  shape, (b) the per-character indexing role (each entry is one
  character's base data pointer), and (c) the motion-system context
  (every consumer is in the motion lookup path).
* Confidence **medium-high** because:
  1. The initialisation gives us concrete addresses, anchoring the
     "upper-RAM data region" interpretation.
  2. The per-character indexing is unambiguous in 4 read sites.
  3. The adjacent slots `D_800A3830` / `D_800A3864` and the matching
     `player_rob_Init` heap_base argument together establish this is
     the per-character data-region table.
* Not full **high** because: the exact array length (3? 8? 16?)
  isn't visible; the indexing constants (`arg0[0x4A/2]`, `v1`, `v0`)
  could conceivably index into something larger that overlaps with
  D_800A3864 etc. -- a struct-vs-array decision that needs more
  decomp.
