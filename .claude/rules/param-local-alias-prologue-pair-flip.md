# Flip cc1's prologue save+def pair order via local-alias capture of params in target order

## Symptom

A function carries a single regfix `reorder` rule on the prologue that swaps
the save+def pairing order — typically a 4-instruction permutation reshuffling
cc1's natural `[sw $s1; move $s1, $a0; sw $s0; move $s0, $a1]` into target's
`[sw $s0; move $s0, $a1; sw $s1; move $s1, $a0]` (low-numbered callee-save
saved+initialized FIRST instead of the natural HIGH-to-LOW order).

```
func_X: reorder 3,4,1,2 @ 1-4
```

`canonical` routes `C`, `sandbox --disable all` reads a small number (typically
4), the diff is purely the prologue pair-order swap, and no body restructure
has closed it. The function's signature is typically `void func(T0 *arg0,
T1 *arg1)` where arg1 (the second param) is the one referenced FIRST in the
body (which is why RA correctly assigns arg1 -> $s0 and arg0 -> $s1; the
*assignment* matches target, just the *save+def order* doesn't).

## Cause

cc1's `mips.c:save_restore_insns` iterates `GP_REG_LAST -> GP_REG_FIRST`
descending (line ~4680: `for (regno = GP_REG_LAST; regno >= GP_REG_FIRST;
regno--)`), emitting `sw $sN` for each saved callee-save in HIGH-to-LOW
order. The matching arg-init `move $sN, $aM` RTL is generated by
`expand_function_start` in `function.c` in DECL_ARGUMENTS order (arg0
first, then arg1). The scheduler then pairs each `sw $sN` with its
corresponding `move $sN, $aM` and emits them HIGH-to-LOW.

The prologue saves+inits are NOT visible to `sched1` — they're emitted
post-flow by `expand_prologue`. So scheduler-level levers (priority bumps,
chain extensions, reference-count manipulation) **CANNOT** reorder them.
The save order is essentially fixed by the iteration direction in
`save_restore_insns`.

Target's `[sw $s0; move $s0, $a1; sw $s1; move $s1, $a0]` order requires
either patching mips.c (forbidden per [[no-compiler-divergence]]) or finding
a way that makes cc1 use **DIFFERENT pseudos for the callee-save targets** —
ones whose creation order is reversed from arg pseudos.

## The fix — capture params into local aliases declared in target's pair order

Replace direct parameter references with local-alias captures, declaring
the alias for the SECOND parameter (the one target saves FIRST) BEFORE
the alias for the FIRST parameter:

```c
void func_8007C2A0(s32 *out, Rect *r)
{
    Rect *_r = r;       /* arg1 captured FIRST -- gets local pseudo with low LUID */
    s32 *_out = out;    /* arg0 captured SECOND -- gets local pseudo with higher LUID */
    /* ... body uses _r and _out instead of r and out ... */
}
```

cc1's `expand_function_start` still creates the arg pseudos in DECL_ARGUMENTS
order. But now the **assignments to the local aliases** are explicit C
statements that emit their own `(set local_pseudo arg_pseudo)` RTL insns
in the order they appear in source. The local aliases are SEPARATE pseudos
from the arg pseudos; they get their own allocation.

The scheduler picks the local-alias `move` for `_r` (the second-declared
local but FIRST in source) ahead of `_out` (because `_r`'s init RTL has a
lower LUID — emitted first in source order). And since the body uses `_r`
heavily before `_out`, RA assigns `_r -> $s0` (low) and `_out -> $s1` (high).

Then `expand_prologue` emits `sw $s0; move $s0, $a1; sw $s1; move $s1, $a0`
matching target. The regfix `reorder` rule becomes unnecessary; `retire`
drops it; full SHA1 == oracle.

## Why this works (the mechanism, clearly)

The `expand_function_start` creates ARG PSEUDOS in DECL_ARGUMENTS order.
When the function body directly references the args (`out[1] = ...; r->x`),
the SAME arg pseudos serve as the callee-save destinations. RA can't avoid
the HIGH-to-LOW save iteration over them.

When you capture args into LOCALS (`Rect *_r = r;`), GCC generates a fresh
RTL insn `(set local_pseudo (arg_pseudo))` for each alias. The arg pseudos
(originally allocated to callee-saves via DECL_ARGUMENTS order) get DCE'd
once their values flow into the locals. The LOCALS are the pseudos that
need callee-saves — and the LOCALS were declared in target's pair order, so
their RTL emission order matches target's save+def pair order.

The key trick: **the C declaration order of the aliases controls the LUID
of their `move` RTL insns**, and `save_restore_insns` pairs each save with
its highest-priority init insn — which the local-alias trick puts in target
order.

## Confirmed case — func_8007C2A0 (display.c, 2026-06-01)

Queue-parked item (commit `2b3ccfe`), distance 4, single regfix
`reorder 3,4,1,2 @ 1-4` on the prologue pair swap. Prior session tested
9 levers (local alias of `r` only, K&R signatures, dead-store extensions,
3rd callee-save, all-temp-stores) — all stuck at score 4. None of them
declared BOTH locals with the target-order pairing.

This session: declaring `Rect *_r = r;` BEFORE `s32 *_out = out;` and
referencing the locals throughout the body -> `sandbox --disable all` 4 -> 0
on first try; `retire` dropped the regfix rule; SHA1 == oracle.
100% pure C.

The lever is also applicable to the twin function `func_8007C4B8` (identical
signature `(s32 *out, Rect *r)`, identical body shape, identical prior
`reorder 3,4,1,2 @ 1-4` rule).

## When this rule applies

Look for the cluster of preconditions together:
- Function carries a SINGLE regfix `reorder` rule on the prologue indices
  (typically `reorder 3,4,1,2 @ 1-4` or similar 4-instruction permutation).
- `canonical` -> C, `sandbox --disable all` reads small (typically 4).
- The diff is purely the prologue pair order (sw+move sequence reversed).
- The function has 2+ params where the SECOND param is referenced FIRST in
  the body (causing RA to correctly assign arg1 -> $s0, but save order to
  remain HIGH-LOW).

If the diff is in the body (not just the prologue), check the score after
applying this lever — body-level diffs would remain and the score won't
drop to 0.

## When this rule does NOT apply

- The diff is a register rename (allocation issue), not a save-order issue
  -> see [[register-alloc-pure-c]].
- The function has only ONE param or the param order in target matches
  cc1's natural emission -> no save-order problem to solve.
- Adding the local aliases regresses the score (the body's scheduling
  becomes sensitive to the alias pseudo's lifetime — check sandbox after
  each local added).
- The `goto-end-prologue-delay-slot` pattern fits ([[goto-end-prologue-delay-slot]])
  — that's a different scenario (early-exit makes the prologue land in a
  delay slot).
