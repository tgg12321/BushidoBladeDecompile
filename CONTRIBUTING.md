The Bushido Blade 2 decomp is a long-lived, queue-driven project. Most contributions are one-function-at-a-time matching work: pick the top item from `engine/queue.json`, write its C body, and iterate until the build still SHA1-matches with your function as real C instead of assembly.

This document covers the contribution model, the per-function lifecycle, the tooling, and the PR conventions. If you're trying to set up a build instead, start with [`BUILD.md`](BUILD.md).

## Before you start

You need a working, SHA1-verified build. From the project root in a WSL or Linux shell:

```bash
source .venv/bin/activate
make clean-check         # should end with "OK: bb2 matches!"
```

If you don't see `OK: bb2 matches!`, fix the build first. Pulling work from `WORK_QUEUE.md` while the baseline is broken will produce false-positive matches (you can't tell if your edit broke the match if the baseline was already broken).

## How work is selected

There is one canonical, ordered queue of remaining work: [`WORK_QUEUE.md`](WORK_QUEUE.md). It is regenerated by `bash tools/dc.sh refresh-queue` and contains three pullable sub-queues:

| Queue | Pull command | Contents |
|---|---|---|
| **Active decomp** | `dc.sh next` | Functions still implemented as inline `__asm__()` blocks in `src/*.c`. The main backlog. |
| **Structural split** | `dc.sh next-structural` | Functions that need source-file or rodata splitting before they can be matched (mostly resolved). |
| **Asmfix retirement** | `dc.sh next-asmfix` | Functions currently bridged with `replace_with_asmfile` in [`asmfix.txt`](asmfix.txt). Need to be rewritten as real C. |

**Do not hunt for "easier" functions.** The queue is in (loose) complexity order. Functions at the top are first, regardless of whether they look intimidating. The project's working rule (THE HARD RULE) is:

> Once a function is selected, work it end-to-end until it matches in 100% pure C (or in an authorized canonical inline-asm form) with a green SHA1. Don't table it, don't skip it, don't switch to a different function because you're stuck. "Stuck" means switch **technique**, not target.

This rule exists because the project accumulated 209 `replace_with_asmfile` bridges across rounds of "this one's too hard" before THE HARD RULE was added. Bridging is now an explicit-authorization step, not a default.

To preview the top of the queue without claiming a function:

```bash
bash tools/dc.sh next 5            # top 5, dry-run
bash tools/dc.sh next-asmfix 5
```

## The active-marker hook

When you pull a function with `bash tools/dc.sh next`, the script writes the function name to `.bb2_active_func`. The `grind_check.sh` **Stop** hook then keeps you on that function: it refuses to let a Claude Code turn end while the active function is still unmatched (unless you `dc.sh release` it).

> **Deprecation (2026-05-22):** the older `tools/hooks/active_func_guard.sh` **PreToolUse** hook — which *refused* `git commit` until match, blocked `git checkout`/`restore`/`reset` of in-progress build files, and blocked `dc.sh next*` while active — has been **deprecated** (unwired). It blocked natural progress: trying new approaches and restoring old attempts when work went awry. You are now free to commit, revert, and experiment mid-work; staying on-task is enforced only at turn-end by `grind_check`. Commit-time **cheat auditing** (programmatic `audit_asm_cheats.py --check-new` + the adversarial LLM auditor) is unaffected — it lives in `tools/hooks/commit_audit_guard.sh`.

Clear the marker via:

| Path | When to use it |
|---|---|
| `bash tools/dc.sh refresh-queue` | The normal path after a match — regenerates the queue and clears the marker. (The old auto-clear on a matching commit was part of the now-deprecated `active_func_guard` hook.) |
| `bash tools/dc.sh release` | Explicit user-driven abandonment. Requires typing the function name to confirm. Don't use this casually. |

You can preview the top of the queue (`dc.sh next 5`) while a function is active; only the top-1-pull form (`dc.sh next` with no count) sets the marker.

## Per-function workflow

The typical lifecycle for one function:

```
dc.sh next --with-context        # pull + auto-run agent-brief
... read the asm, the source, neighbors, Kengo references ...
dc.sh preflight <func>           # one-screen brief: bridge state, size, hints
dc.sh attempt <func>             # full mechanical pipeline (smart + permute + gen-regfix)
... if MATCHED, dc.sh inline-replace <func> auto_matches/<func>.c
... if NEAR_MISS / HARD, iterate manually ...
dc.sh build-active <func>        # incremental rebuild + bridge-aware verify
dc.sh diff-summary <func>        # quick verdict if not matching yet
... edit src/<file>.c, write regfix rules, etc. ...
dc.sh verify <func>              # binary verify
dc.sh verify --all               # sibling regression check
git add src/<file>.c regfix.txt  # stage your changes
git commit -m "match func_..."   # hook re-verifies, then commit succeeds and marker clears
bash tools/dc.sh refresh-queue   # regenerate WORK_QUEUE.md (function falls off)
```

### Pulling a function

```bash
bash tools/dc.sh next --with-context
```

This claims the top of the queue, sets `.bb2_active_func`, and runs `tools/agent_brief.py` to dump:

- The original assembly (`asm/funcs/<func>.s`)
- The current C location and any existing source body
- Classifier output (size, blockers, GTE usage, BIOS hints)
- Recipe suggestions (the recipe library was archived 2026-05-26; technique knowledge now lives in `docs/MATCHING.md` and `.claude/rules/`)
- Kengo PS2 sister-function (when a credible match exists)
- Adjacent functions in the same file

### Writing the C body

The original was compiled with **GCC 2.7.2** (PsyQ/SN Systems fork) at `-O2 -G0 -funsigned-char`. Matching means producing C that compiles to identical machine code. The compiler has known quirks:

- Variable declaration order in many cases determines register allocation order (`$s0, $s1, ...`).
- Optimizer passes (CSE, LICM, strength reduction, dead-code elimination) are aggressive enough that the C you write rarely matches verbatim what you see in the disassembly. Match the *output*, not the source.
- Some constructs cannot be expressed in C 2.7.2 at all (custom calling conventions, certain GTE primitives, BIOS trampolines). These are authorized in [`inline_asm_canonical.txt`](inline_asm_canonical.txt) with one-line justifications. Don't add to that list without explicit authorization.

[`docs/MATCHING.md`](docs/MATCHING.md) has a distilled playbook of C-side techniques, the post-pipeline `regfix.txt` rewrite layer, and named recipes for common patterns.

### Integrating a match

The successful permuter/manual match is typically a standalone `base.c` in `permuter/<func>/` or `auto_matches/<func>.c`. To integrate:

```bash
bash tools/dc.sh inline-replace <func> auto_matches/<func>.c
```

`inline-replace` does two important things:

1. Splices the matched function body into `src/<file>.c`, replacing any existing inline-asm block.
2. Runs `dc.sh caller-audit <func>` and **blocks the integration** if your function's signature doesn't match the maximum-observed argument count among callers. (Skipping this audit causes cascade-regressions in unrelated sibling functions — GCC dead-codes the extra-argument computations in callers, changing their bytes.)

If `caller-audit` blocks: widen the declaration to the observed arity, using `s32 unused` for arguments you don't reference. Don't override the audit.

### Verifying

Three layers of verification matter:

```bash
bash tools/dc.sh verify <func>         # binary diff one function
bash tools/dc.sh verify --all          # SHA1 short-circuit + per-function scan
bash tools/dc.sh verify --clean        # full clean rebuild + verify --all
```

- `verify <func>` is fast but bridge-blind: it always reports MATCH for a function that has an active `replace_with_asmfile` line in `asmfix.txt`. For functions you're retiring, use `dc.sh verify-c <func>` instead.
- `verify --all` short-circuits on the whole-binary SHA1; if it passes, every function matches. ~0.2 seconds when green.
- `verify --clean` is the trustworthy long-form: wipes `build/`, rebuilds everything, then verifies. Run this before commits that involve multiple register-asm pins, more than ~3 regfix rules, or sensitive scheduling tricks. Catches the build-cache regression trap.

### Committing

The active-marker hook gates `git commit`. It runs `dc.sh verify <active>` and only allows the commit if the result is `MATCH`. On success, the marker is auto-cleared.

```bash
git add src/<file>.c regfix.txt asmfix.txt   # whatever you touched
git commit -m "match <func>: brief one-line description"
bash tools/dc.sh refresh-queue               # drops the matched function from WORK_QUEUE.md
```

Commit message style (from `git log --oneline`):

```
text1b: match func_8004C388 (xyz+packed-rgb midpoint averager, ...)
auto-repair: detect SHA1-silent pair-shift drift via MISMATCH line
text1b: authorize func_800526A0 canonical inline-asm (GTE LZCS/LZCR primitive)
display.c: match func_8007ED6C (GTE 3x3 wrapper, 8-rule regfix)
```

- Prefix with the affected source file or system (`text1b:`, `tools:`, `auto-repair:`, `docs:`).
- Past tense, lower case.
- One short sentence about what changed and why it matters (the matching strategy, the recipe used, the byte/score the change closes).
- Reference the function name and any relevant address.

## Escalation ladder when stuck

A function isn't matching after your first attempt. The right next step depends on the **penalty profile** reported by `bash tools/dc.sh debug <func_dir>` or `bash tools/dc.sh diff-align <func>` — not on intuition. The ladder, in order:

1. **C-level structural variants.** Different loop forms, intermediate variables for subtraction order, declaration position. The permuter (`bash tools/dc.sh permute <func>`) explores these automatically.
2. **Register-asm pinning** (callee-saves only). `register T x asm("$s2");` is a hint to GCC's register allocator. Useful for closing the last ~5 register-naming diffs once structure is correct. Leaf temp regs (`$t0`–`$t9`) are dangerous — pinning causes hoisted sign-extensions and altered delay-slot fills.
3. **Long permuter runs.** `bash tools/dc.sh permute <func> --max-time 1800` runs the permuter for 30 minutes. Overnight runs (3600s+) are acceptable.
4. **regfix at the assembly stream.** `regfix.txt` rewrites maspsx output before the assembler sees it. Use `bash tools/dc.sh regfix-suggest <func>` to auto-emit rules from the diff. Manual rules: `bash tools/dc.sh add-regfix <op> <func> <args>` (pre-validates against `dump_text_indices`).
5. **Compound regfix layering** — 10+ rules across multiple blocker classes (register cycle + scheduling + frame mismatch). Not unusual.
6. **Named recipes** — common patterns (LICM unhoist, call-loop family, early-exit alias breaker, GTE 3x3 wrapper) are documented in [`docs/MATCHING.md`](docs/MATCHING.md) and as path-scoped technique docs in `.claude/rules/`. (The machine-readable recipe library + its auto-suggest/apply tooling were archived 2026-05-26; recognize the symptom from the docs and write the C / regfix yourself.)
7. **New transformation pass** — extend `prologue_fix.py`, add a pipeline stage, write a new regfix op. The project's pipeline is open-ended; building new tooling when stuck is expected.

**After a match — register what you learned.** If the match revealed a reusable codegen pattern or a non-obvious gotcha, record it *before committing* (this is the "Register findings" step in [`CLAUDE.md`](CLAUDE.md)): a reusable pattern → a new/updated path-scoped doc in `.claude/rules/<slug>.md` (auto-loads on matching source reads, and the metrics layer fingerprints it as a technique `slug`); a function-specific fact → a `memory/` entry. There is no longer a recipe-capture tool — this human-written record is the durable one.

The full playbook with penalty-profile → technique tables, C-side optimization defeats, regfix syntax reference, and named recipe documentation is in [`docs/MATCHING.md`](docs/MATCHING.md).

What stuck does **not** mean:

- A reason to skip the function.
- A reason to bridge with `replace_with_asmfile`.
- A reason to leave inline `__asm__()` as the final answer.

If you genuinely believe a function is unreachable in pure C, open an issue with the asm, the techniques you tried, and the penalty profile — that's a project-wide conversation, not a per-function shortcut.

## Bridge / asmfix retirement workflow

Some functions are currently bridged: their entry in `asmfix.txt` reads `funcname: replace_with_asmfile "asm/funcs/funcname.s"`, which substitutes raw hand-disassembled MIPS for the C body at build time. They count as **not done** and live on the asmfix retirement queue (`bash tools/dc.sh next-asmfix`).

To retire one:

```bash
bash tools/dc.sh next-asmfix                # claim a bridge function
bash tools/dc.sh preflight <func>           # check bridge state, signatures, hints
bash tools/dc.sh retire <func>              # comments out the asmfix bridge with `# RETIRE:`
... write the real C body in src/<file>.c ...
bash tools/dc.sh build-active <func>        # incremental build + bridge-aware verify
bash tools/dc.sh verify-c <func>            # honest verification (refuses if bridge still active)
git commit -m "retire <func>: ..."
```

`dc.sh retire` does the right thing automatically — it comments out the bridge line and sets the active marker. The commit hook refuses to allow commits while a bridge is half-restored.

**Use `dc.sh verify-c` not `dc.sh verify` for retirement work.** Plain `verify` reads bytes at the function's address in `build/bb2.exe`, which for a bridged function are always the bridge's bytes regardless of what's in your C body — false MATCH. `verify-c` refuses to run when an active bridge is present.

When the C body genuinely matches, `bash tools/dc.sh purge-retirements` cleans up the `# RETIRE:` markers in `asmfix.txt`.

## Adding canonical inline-asm

A few categories of functions cannot be expressed in C 2.7.2:

- **PSX BIOS trampolines** — 3-instruction sequences that tail-call a fixed kernel vector with a function number in `$t1` in the delay slot. GCC has no construct for this.
- **GTE primitives** — direct coprocessor 2 ops (`mtc2`, `mfc2`, `ctc2`, `cfc2`, `cop2`, `lwc2`, `swc2`, `.word 0x4xxxxxxx`) for the geometry transformation engine. PsyQ's `include/gte.h` wraps the common ones in C macros; the raw primitives need inline asm.
- **Custom calling conventions** — functions where the caller passes data via `$s0` without prologue saving, or other ABI departures from the standard PsyQ calling convention.
- **Hand-coded math kernels** — sin/cos rotation kernels, 3-axis Euler rotation, etc. that show strong evidence of hand-written assembly (uniform multu pacing, front-loaded loads, INT_MIN guard idioms).

These are tracked in [`inline_asm_canonical.txt`](inline_asm_canonical.txt) with per-function one-line justifications. **Adding a function to that list requires explicit authorization** from a maintainer (open an issue with evidence — the `bash tools/dc.sh scan-hand-coded --single <func>` and `bash tools/dc.sh memory-check <func>` tools surface the relevant signals). Do not add to the list pre-emptively because a function is hard.

## Testing and verification

There is no traditional test suite (the test is "does it match"). The pipeline tooling has its own self-tests:

```bash
python3 tools/test_auto_drift_rollback.py
python3 tools/test_maspsx_stripping_scanner.py
python3 tools/test_regfix_reorder_idx0.py
```

The real verification gates:

| Layer | Command | Cost | What it catches |
|---|---|---|---|
| Build | `make` | ~3–10s incremental | Compilation errors, regfix syntax errors, link errors |
| Per-function binary | `dc.sh verify <func>` | <1s | Byte differences vs `asm/funcs/<func>.s` |
| Whole-binary SHA1 | `dc.sh verify --all` | ~0.2s (cached) | Any byte difference vs original `SLUS_006.63` |
| Clean rebuild | `dc.sh verify --clean` | ~1–2 min | Build-cache regressions hidden by stale `.o` files |
| Sibling regression | `dc.sh post-match-validate <func>` | ~5s | `.L<N>` label-drift breaking other functions in the same file |
| Regfix rules | `make validate` | ~10s | Stale or no-op regfix rules from upstream label changes |

Always run `dc.sh verify --all` after every match. Run `dc.sh verify --clean` before commits that touch register-allocation-sensitive code.

## PR conventions

This project lives on the `main` branch on GitHub.

| Convention | Detail |
|---|---|
| No direct commits to `main` from contributors | Open a PR against `main` from a topic branch. |
| Branch naming | `<scope>/<short-description>`, e.g. `text1b/match-func-8004C388`, `tools/regfix-suggest-fix`, `docs/build-md-rewrite`. |
| One match per PR (preferred) | Easier review, easier rollback if a sibling regression slips through. Stacks of 2–3 closely-related matches are acceptable when they share a recipe. |
| Each commit must SHA1-match | The hook enforces this for committers; reviewers verify the same. Don't squash multiple unmatched WIPs into "match" — that hides the work. |
| Include the function name in the PR title | `match func_8004C388: xyz+packed-rgb midpoint averager` |
| In the PR body, summarize the technique | What recipe applied (if any), what register-asm pins were needed, what regfix rules were added, what `caller-audit` flagged. Future maintainers reading `git log` need this. |
| If you touched `regfix.txt`, `asmfix.txt`, or pipeline tools | Mention which functions are affected and run `make validate` + `dc.sh verify --all` before pushing. |
| If you renamed symbols (in `named_syms.txt`) | Run `python3 tools/audit_named_syms_orphans.py --apply` to clean up shadows. |
| Don't push the build cache | `build/` is gitignored. Don't add `*.o`, `*.elf`, `*.exe`, or `*.bin` files. |
| Don't push the disc image | `*.bin`, `*.cue`, `*.iso` are gitignored. |

## Code of conduct

Be kind. Decompilation is collaborative archaeology; we're all trying to understand a 1998 build with limited evidence. Disagreements about naming, technique, or strategy are normal — discuss them openly. Don't claim functions you can't finish; don't bridge functions to avoid hard work; don't push commits that break the build.

If your contribution interacts with the Claude Code agent (the project uses an LLM coding agent for most matching work), follow the operating contract in [`CLAUDE.md`](CLAUDE.md). In particular: don't manually edit `.bb2_active_func`, `.bb2_iter_log/`, `WORK_QUEUE.md`, or any of the queue-generation outputs.
